分散システムと OAuth2 アーキテクチャにおいて、Identity Provider (IdP) は署名鍵を更新しながら、ダウンストリームサービス（リソースサーバー、ゲートウェイ）がシームレスにそれを感知し、制御を失わないようにするにはどうすればよいでしょうか？これはシステムの可用性とセキュリティに関わる核心的な問題です。

本文では、JWKS 動的発見の基本原理から始め、鍵のライフサイクル状態遷移、多インスタンス環境下での状態整合性の課題、そしてリフレッシュトークンローテーションにおける "Grace Replay" 防護メカニズムまで、完全な技術的展望を解説します。

---

## 1. 鍵ローテーション：ダウンストリームはどうやって「シームレスに」同期するのか？

IdP が署名用の秘密鍵をローテーションした際、ダウンストリームサービスが古い公開鍵を保持したままだと、正当なトークンを無効と判定してしまいます。これが鍵ローテーションが直面する最も基本的な問題です。**署名側の鍵は変わったが、検証側がそれを知らない**という状態です。

### 1.1 JWKS メカニズムのコア思想

業界標準のソリューション（RFC 7517 / RFC 7518）は、3つの要素でこの問題を解決します：

- **kid (Key ID)**：IdP は各 JWT のヘッダーに `kid` フィールドを含め、「このトークンはどの鍵で署名されたか」を識別させます。`kid` 自体は鍵の材料を含まず、単なる識別子（UUID、タイムスタンプのハッシュ、または任意のユニークな文字列）です。
- **JWKS エンドポイント**：IdP は標準的なエンドポイント（本プロジェクトでは `/.well-known/jwks.json` または `/oauth2/jwks`）を公開し、現在有効なすべての公開鍵のセットを返します。このレスポンスは、JWK (JSON Web Key) の配列を含む JSON オブジェクトです。
- **動的プル（Dynamic Pull）**：ダウンストリームサービスは検証時にシンプルなロジックを実行します。まず、ローカルキャッシュから `kid` に対応する公開鍵を探します。ヒットすればそのまま検証します。ミス（キャッシュにない）した場合、IdP が鍵を更新した可能性があると判断し、JWKS エンドポイントに HTTP リクエストを送信して最新の鍵セットを取得し、ローカルキャッシュを更新してから再度検証を試みます。

このメカニズムの巧みな点は、ダウンストリームサービスに事前の設定や通知が不要なことです。鍵の同期は検証プロセス中の `kid` ミスによって完全に駆動されます。

本プロジェクトにおける JWKS エンドポイントの実装（`idp-server/internal/application/oidc/service.go`）は以下の通りです：

```go
func (s *Service) JWKS(ctx context.Context) (*JSONWebKeySet, error) {
	_ = ctx
	// 鍵プロバイダーが注入されていない場合は空のセットを返す
	if s.keys == nil {
		return &JSONWebKeySet{}, nil
	}
	// PublicJWKS() は KeyManager から現在有効なすべての公開鍵（アクティブおよび廃止予定のものを含む）を抽出します
	return &JSONWebKeySet{Keys: s.keys.PublicJWKS()}, nil
}
```

ここで重要なのは `PublicJWKS()` が返すのは「現在アクティブな鍵」だけではなく、「廃止予定だがまだ検証に有効な鍵」も含まれるという点です。この挙動こそが、次節で述べるスムーズな移行を可能にしています。

### 1.2 スムーズな移行戦略（Grace Period）

JWKS の動的プルメカニズムは「同期できるか」という問題を解決しますが、「いつ同期するか」については慎重な設計が必要です。もし IdP が「新しい鍵での署名開始」と「新しい公開鍵の公開」を同時に行うと、ダウンストリームのキャッシュが更新されるまでの間、新しい `kid` を持つトークンを受け取ったすべてのダウンストリームが一斉に JWKS をプルし始めます。

これは通常は問題ありませんが、高トラフィック環境下では、大量のリクエストが同時に JWKS プルをトリガーし、IdP に瞬間的な負荷をかける「群衆効果（Thundering Herd）」を引き起こす可能性があります。さらに深刻なのは、JWKS エンドポイントがその瞬間にたまたま利用不可（ネットワーク不安定、デプロイ中のリスタート）だった場合、これらのリクエストはすべて検証に失敗するということです。

推奨されるアプローチは、ローテーションを4つの段階に分けることです：

1. **事前公開（Pre-publish）**：新しい公開鍵を JWKS エンドポイントに追加しますが、IdP はまだ古い秘密鍵で署名を続けます。これにより、ダウンストリームは通常のキャッシュ更新サイクルの中で「自然に」新しい鍵を取得できます。`kid` ミスが発生しないため、一斉プルも起きません。
2. **観察期（Soak）**：ダウンストリームのほとんどがキャッシュを更新するまで待ちます。この時間は通常、ダウンストリームの JWKS キャッシュの TTL より長く設定すべきです。キャッシュ TTL が 24 時間なら、この期間も最低 24 時間です。
3. **正式切り替え（Activate）**：IdP が新しい秘密鍵で新しいトークンの署名を開始します。この時点では JWKS には新旧両方の公開鍵が含まれているため、どちらのトークンも検証可能です。
4. **旧鍵の廃止（Decommission）**：古いトークンがすべて期限切れになるのを待ってから、JWKS から古い公開鍵を削除します。

この4段階の核心思想は、**先に検証側を準備させてから、署名側を切り替える**ということです。逆にすると、署名が変わった瞬間から検証が追いつくまでの間、すべてのリクエストがリスクにさらされます。

本プロジェクトの `idp-server/internal/infrastructure/crypto/key_rotation.go` では、設定によってこのロジックを実現しています：

```go
func rotateKey(ctx context.Context, repo rotationRepository, cfg RotationConfig, force bool) (bool, error) {
	// ... デフォルト値の設定
	if cfg.RotateBefore <= 0 {
		cfg.RotateBefore = 24 * time.Hour // 24時間前に新しい鍵を生成し、JWKSに公開
	}
	if cfg.RetireAfter <= 0 {
		cfg.RetireAfter = 24 * time.Hour  // 古い鍵は置換後も24時間は検証用に保持
	}
	// ...
}
```

### 1.3 鍵のライフサイクル状態遷移

上記の4段階を正確に理解するには、1つの鍵がどのような状態遷移を辿るかを把握しておく必要があります。本プロジェクトにおける鍵の状態は以下の通りです：

```
[生成] ──→ [Standby] ──→ [Active] ──→ [Retired] ──→ [削除]
              │               │              │
              │               │              └─ JWKS に公開鍵あり（検証用）
              │               │                 署名には使われない
              │               └─ JWKS に公開鍵あり、署名にも使用中
              └─ JWKS に公開鍵あり、署名にはまだ使われない（Pre-publish 状態）
```

各状態の意味は以下の通りです：

- **Standby**：鍵ペアは生成済みで、公開鍵は JWKS エンドポイントに含まれているが、まだ署名には使われていない。ダウンストリームがキャッシュを更新して「この kid を知っている」状態になるまでの猶予期間。これが `RotateBefore` パラメータが制御する区間です。
- **Active**：現在の署名鍵。すべての新しいトークンはこの鍵で署名されます。JWKS に含まれ、署名にも検証にも使用されます。
- **Retired**：次の鍵が Active になった後の旧鍵。JWKS にはまだ含まれ、既存のトークン（まだ有効期限内のもの）の検証に使われます。しかし、新しいトークンの署名には使いません。これが `RetireAfter` パラメータが制御する区間です。
- **削除**：`RetireAfter` の期間が経過し、この鍵で署名されたすべてのトークンが期限切れになったと見なされた後、JWKS から完全に削除されます。

## 2. 多インスタンスデプロイにおける整合性の課題

JWKS メカニズムはアーキテクチャとしては優雅ですが、実務の実装において見落とされがちな問題があります。それは、**IdP 自体が多インスタンスでデプロイされている場合、各インスタンス間の JWKS 内部状態が不一致になる可能性がある**ことです。

### 2.1 問題の根本原因

典型的な実装（例えば `StartRotationLoop()` によるバックグラウンドタスク）では、各インスタンスが独立してタイマーを実行し、共有ストレージ（DB 等）をチェックして鍵をメモリ上の `KeyManager` にロードします。

ローテーション自体は分散ロック等により1つのインスタンスでのみ行われますが、他のインスタンスの `KeyManager` 更新はそれぞれのタイマーに依存します。インスタンス A が T=0 でローテーションを完了しても、インスタンス B のタイマーが次に動くのが T=55分後であれば、その間 B は新しい鍵を知りません。

この問題の本質は**分散キャッシュの無効化**です。各プロセスのメモリは独立したキャッシュの複製であり、相互通知のメカニズムがありません。

### 2.2 解決策：Redis Pub/Sub + Ticker によるバックアップ

有効な解決策は、Redis Pub/Sub をインスタンス間の通知メカニズムとして導入することです：

1. **発行（Publish）**：いずれかのインスタンスがローテーションを完了したら、Redis チャネルに通知を送信します。
2. **購読（Subscribe）**：すべてのインスタンスがこの通知を待ち受け、受信したら即座にメモリ上の `KeyManager` をリロードします。

本プロジェクトでは `KeySyncBroadcaster` (`idp-server/internal/infrastructure/crypto/key_sync.go`) を通じてこの同期を実現しています。ここではバースト的な通知を抑制するためのデバウンス（Debounce）処理も含まれています：

```go
func (b *KeySyncBroadcaster) Subscribe(ctx context.Context, manager *KeyManager, repo rotationRepository, workingDir string) {
	// ... (初期化処理)
	go func() {
		pubsub := b.rdb.Subscribe(ctx, b.channel)
		defer func() { _ = pubsub.Close() }()
		// ...
		for {
			select {
			case _, ok := <-ch:
				if !ok { return }
				if !pending {
					debounce.Reset(500 * time.Millisecond) // 500msのデバウンス
					pending = true
				}
			case <-debounce.C:
				pending = false
				// 通知を受け取った後、DBから最新の鍵セットをフルロードする
				refreshed, err := LoadKeyManagerFromRepository(loadCtx, repo, workingDir)
				if err == nil {
					manager.ReplaceWith(refreshed) // メモリ上の状態をアトミックに置換
				}
			}
		}
	}()
}
```

### 2.3 ReplaceWith のアトミック性

メモリ上の状態置換は `sync.RWMutex` を用いて、リクエスト処理中の中途半端な状態が外部から観測されないよう保証しています：

```go
// idp-server/internal/infrastructure/crypto/key_manager.go

func (m *KeyManager) ReplaceWith(other *KeyManager) {
	if other == nil { return }
	// 新しい状態のクローンを作成
	other.mu.RLock()
	cloned := make(map[string]*managedKey, len(other.keys))
	for kid, record := range other.keys {
		cloned[kid] = &managedKey{ /* ... */ }
	}
	other.mu.RUnlock()

	// 書き込みロックを取得して一括置換
	m.mu.Lock()
	defer m.mu.Unlock()
	m.activeKID = other.activeKID
	m.keys = cloned
}
```

## 3. 制御不能？リフレッシュトークンは最後の救済策

ダウンストリームサービスが公開鍵をハードコードしていたり、極端に長いキャッシュを持っていたりして同期に失敗した場合でも、IdP にはまだ防衛線があります。それが、**短寿命のアクセストークン + リフレッシュトークンローテーション**です。

アクセストークンの有効期限を 15 分に設定すれば、たとえダウンストリームのキャッシュが完全に壊れていても、15 分以内に同期が回復します。これは、新しいトークンが発行されるたびに、最新の秘密鍵と新しい `kid` が使用され、それが検証側の `kid` ミス（＝JWKS リフレッシュ）を強制するためです。

## 4. 高度なセキュリティ：リフレッシュトークンローテーションと Grace Replay

セキュリティ要件が高い環境では、リフレッシュトークンは**使い捨て（One-time use）**であるべきです。

### 4.1 競合状態（Race Condition）の課題

ネットワーク断等により、クライアントが「リフレッシュ要求は成功したがレスポンスを受け取れなかった」場合に再試行すると、IdP はすでに古いトークンを「使用済み」として無効化しているため、拒否されてしまいます。

### 4.2 Grace Replay メカニズム

これを解決するために、「短い猶予期間（Grace Period）」を設けます。本プロジェクトでは `idp-server/internal/application/token/service.go` で以下のように実装されています：

```go
func (s *Service) exchangeRefreshToken(ctx context.Context, input ExchangeInput) (*ExchangeResult, error) {
	// ...
	if s.tokenCache != nil {
		// 第1段階チェック：キャッシュから直近の成功結果を取得。
		replay, _ := s.tokenCache.CheckRefreshTokenReplay(ctx, oldSHA, strings.TrimSpace(input.ReplayFingerprint))
		if result := refreshReplayToExchangeResult(replay); result != nil {
			return result, nil // Grace Replay ヒット：前回の成功レスポンスをそのまま返す
		}
	}

	// ... (DBでのローテーション処理) ...

	if err := s.tokens.RotateRefreshToken(ctx, oldSHA, now, newRefresh); err != nil {
		// 第2段階チェック：DBでの更新が競合で失敗した場合、再度キャッシュから復元を試みる
		if replayResult, replayErr := s.tryRefreshTokenGraceReplay(ctx, oldSHA, input.ReplayFingerprint); replayErr == nil && replayResult != nil {
			return replayResult, nil
		}
		return nil, ErrInvalidRefreshToken
	}
	// ...
}
```

この2段階チェックにより、ネットワーク断や並行リクエストによる「偽の失敗」を防ぎながら、厳格な使い捨てポリシーを維持しています。

## 5. トークンファミリー（Token Family）と連鎖的な失効

リフレッシュトークンローテーションでは、侵害が疑われる場合（猶予期間外に再利用された場合等）、その「家系（Token Family）」全体を無効化する必要があります。

### 5.1 侵害検出のロジック

本プロジェクトでは、Redis キャッシュ層の Lua スクリプト (`idp-server/scripts/lua/check_refresh_replay.lua`) によって、侵害の検出とファミリー全体の即時封禁をアトミックに実行しています：

```lua
-- ... status == "rotated" (既に使用済み) の場合 ...
if now_ts <= grace_until then
    -- 猶予期間内であれば、フィンガープリントをチェックして Grace Replay を許可
    if bind_fp == "" or bind_fp == ARGV[2] then
        local cached = redis.call("GET", KEYS[2]) or ""
        if cached ~= "" then
            return { 1, cached } -- 正常な再試行として成功レスポンスを返す
        end
    end
end

-- 猶予期間外、またはフィンガープリント不一致：侵害と判定
local family_ttl = redis.call("TTL", KEYS[1])
-- ファミリー全体の失効マーカーをRedisにセット
redis.call("SET", family_key, "1", "EX", family_ttl)
-- 該当トークンを「侵害済み (compromised)」としてマーク
redis.call("HSET", KEYS[1], "status", "compromised")
return { -1, "" } -- リクエストを拒否し、連鎖的な失効をトリガー
```

## 6. 実装上の注意点：データベーストランザクション

リフレッシュトークンのローテーションは、DB レベルでアトミックに行われ、「二重使用（Double Spending）」を防ぐ必要があります。

```go
// idp-server/internal/infrastructure/persistence/token_repo.go

func (r *TokenRepository) RotateRefreshToken(ctx context.Context, oldTokenSHA256 string, revokedAt time.Time, newToken *tokendomain.RefreshToken) error {
	tx, _ := r.db.writer().BeginTx(ctx, nil)
	// FOR UPDATE による行ロックで競合を防ぐ
	oldToken, _ := scanRefreshToken(tx.QueryRowContext(ctx, tokenRepositorySQL.rotateFindOldForUpdate, oldTokenSHA256))
	
	// 状態の厳格なチェック
	if oldToken == nil || oldToken.RevokedAt != nil || oldToken.ReplacedByTokenID != nil || !oldToken.ExpiresAt.After(revokedAt) {
		return sql.ErrNoRows // 失敗させることで上層の侵害検出ロジックを動かす
	}

	// 新しいトークンの挿入と旧トークンの無効化を同一トランザクションで実行
	// ...
	return tx.Commit()
}
```

## 7. まとめ

IdP によるダウンストリームの制御力は、単一の機能ではなく、多層的な防衛線によって構築されます：

1. **JWKS + kid + Grace Period**：標準的な自動同期。
2. **KeySyncBroadcaster (Redis Pub/Sub)**：インスタンス間の迅速な同期。
3. **短寿命アクセストークン**：暗黙的な同期トリガー。
4. **Grace Replay**：使い捨てトークンの可用性と耐障害性の両立。
5. **Token Family 連鎖的失効**：異常検知時の迅速なアクセス遮断。

優れたセキュリティアーキテクチャとは、各層が失敗し得ることを受け入れ、必ず次の層がバックアップとして機能するように設計することです。
